# check-spec-structure — the node-shape audit within one project spec

The **check-spec-structure** procedure: audit the **internal node-shape** of one project spec and
emit a finding set for the formation Warden. It is the **intra-spec** successor to the retired
cross-spec tools (`dedupe-specs`/`split-spec`) — now that one project is **one spec**,
structural maintenance happens *inside* a spec, not across many. It is a read-only, advisory
detector (the `../place-node/` family): it computes findings and writes nothing; the Warden decides
and acts (`../../formation/`).

It carries two **deterministic** checks plus two **judgment** arms:

- **untagged-node** (blocking integrity) — a behavioral/reference node whose README declares a
  `spec-type` but carries **no `concept:` tag**, so it never appears in the generated by-concept
  index (`../concept-index/`). This is a real reachability gap; `--check` fails on it.
- **oversized-node** (advisory) — a node whose `.feature` scenario count exceeds the granularity
  threshold. The finding carries a **deterministic shape profile** — the plain-vs-`@rubric`
  scenario counts plus a **soft section-cluster count** (both the box-drawing `# ──` and the dashed
  `# ----` header styles) — but it **prescribes no route**: whether the overflow is **breadth**
  (distinct responsibilities → split) or **depth** (edge-cases per behavior → down-level or
  redesign) is the Warden's call, not the engine's. Advisory only — it never fails `--check`.
- **breadth-vs-depth routing** (judgment) — reading the shape profile to route an oversized node:
  breadth-overflow → a **node split** (the `split-spec` concept, retargeted intra-spec);
  depth-overflow on a **deterministic** artifact-type → **down-level** the edge-cases to runnable
  tests via the scenario→test bridge (`../../mission/verify-scenarios/`), shrinking the frozen
  suite instead of splitting; depth-overflow on an **agent-behavior** type → **redesign or accept**
  (no node-test target). Semantic, not mechanical — the breadth/depth line is not reliably
  deterministic from the corpus's cosmetic section headers, so it is a Warden `@rubric` arm; the
  engine ships the profile, **not** the verdict.
- **intra-spec contradiction** (judgment) — two nodes in the same spec that conflict (the
  `dedupe-specs` reconcile concern, retargeted intra-spec). Semantic, not mechanical → carried as a
  single `@rubric` scenario the Warden judges by hand; the engine ships **no** code for it.

## Use Cases

**Subject** — auditing one project spec's internal node-shape from frontmatter + sibling
`.feature` scenario counts, and emitting a finding set (with severity) for the Warden.
**Non-goals** — it never audits **placement-drift** (concepts legitimately scatter — that is
`../place-node/` + handoff + the Warden), never renders the **by-concept view** (`../concept-index/`),
never checks **gate legality** (`../../authoring/spec-gate/`), and never flags **over-fragmented
folders** or **singleton concepts** (see *Out of scope* below for the rationale).

| Trigger | Inputs | Outcome |
|---|---|---|
| **audit node-shape** (default) — a formation pass needs the structural finding set | the project-spec directory | a finding set: each **untagged-node** (blocking) and **oversized-node** (advisory) naming the node; read-only |
| **check drift** (CI) — a no-regression structural guard | the project-spec directory, check mode | exits **non-zero** iff a **blocking** finding exists, and **writes nothing**; advisory-only findings still exit zero |
| **a node missing a concept tag** | a spec-typed node README with no `concept:` | an **untagged-node** finding — the node is orphaned from the by-concept index (blocking) |
| **a node whose suite exceeds the granularity threshold** | a node + its `.feature`, a threshold | an **oversized-node** finding carrying the node's **shape profile** — plain/`@rubric` counts + section-cluster count — and **no route** (advisory) |
| **route an oversized node** (judgment) | an oversized-node finding + its shape profile | escalated for the **Warden**'s breadth-vs-depth judgment via the `@rubric` arm — split (breadth) vs down-level/redesign (depth); the engine emits no route |
| **two nodes in the spec that contradict** (judgment) | two conflicting nodes | escalated for the **Warden**'s judgment via the `@rubric` arm — the engine emits no deterministic finding |

Every scenario in [`check-spec-structure.feature`](./check-spec-structure.feature) maps to one of
these entry points, to the severity/check-mode rules, or to the determinism/boundary rules that
close this spec.

## Severity — blocking vs advisory

A finding carries a **severity**, and `--check` reads it:

- **blocking** (`untagged-node`) — a structural-integrity defect; `--check` exits non-zero.
- **advisory** (`oversized-node`) — a judgment candidate, not a regression; surfaced in the audit
  for the Warden but **never** fails `--check`. (This is why the engine ships green against a project-spec
  that legitimately holds a large node pending a deliberate split.)

## Out of scope (deferred, with rationale)

- **placement-drift is *not* a deterministic check.** A concept legitimately **scatters** across
  capability folders — re-unifying that scatter is exactly what `../concept-index/` exists for. So a
  "node's concept ≠ its folder" scan false-positives on correctly-placed nodes (e.g. `../../formation/`
  and `../../authoring/backfill-project-spec/` both carry `spec-structure` yet belong where they
  sit). Placement is owned by `../place-node/` (advising a **new** node's home), the mission
  **handoff** (finalizing it), and the **Warden's** judgment — not a recurring mechanical audit.
- **over-fragmented capability folder** — no settled heuristic on a young project-spec; high
  false-positive risk. Revisit once oversized/integrity produce signal.
- **stale / singleton `concept:` value** — a singleton concept is legitimately valid, and surfacing
  where a concept lives **is** the `../concept-index/` by-concept view; including it would duplicate
  that engine.

It also does **not** render the by-concept view (`../concept-index/`) or check gate **legality** —
the `## Use Cases`/`## Subject`/spec-type reconcile is `../../authoring/spec-gate/`'s
`check-spec-state`, fail-closed at the gate; this engine is advisory drift over time.

## Determinism and the write boundary

- **Pure derivation, frontmatter + scenario shape only.** Each deterministic finding is a function
  of node frontmatter and the sibling `.feature`'s **shape** — its scenario count, the plain-vs-`@rubric`
  tag split, and the section-header cluster count — alone. **No node body** (README prose) reaches a
  finding, and rendering twice is byte-identical. The shape profile is data for the Warden's route,
  never a route itself.
- **Writes nothing.** The audit emits findings and edits no file; it owns no lifecycle state. Acting
  on a finding (a split, a reconcile) belongs to the Warden (`../../formation/`) under its own
  self-clear-vs-escalate verdict.
